the vulnerabilities AI writes are not the ones you were warned about

Joakim Nordheim Larssen · 2026 · drawn from the group bachelor thesis with Bendik Hagen and Krister Eriksen

All four frontier models we tested protected against SQL injection unprompted: 8 of 8 samples used parameterized queries. The same four, unprompted, hardcoded secrets and shipped insecure session cookies: 0 of 8 got either right. The textbook vulnerability is solved by default. The operational hygiene that rarely shows up in a published code snippet, secrets in env vars and cookie flags, fails in every sample.

The inversion is the finding, and it points at a fix that is not a better model but a cheaper process control.

This is from a bachelor thesis I wrote with Bendik Hagen and Krister Eriksen at Hoyskolen Kristiania, in collaboration with a small Oslo startup, ToSee AS, graded A in 2026. What follows is the practitioner's version: what we measured, where it agrees and disagrees with the literature, and the part that matters if you run a three-person team shipping AI-generated code.

01

the question worth asking

"Is AI-generated code secure" has been asked enough that the answer is a cliche: roughly one in three samples has a problem. Perry et al. (CCS 2023) found AI-assisted participants wrote SQL-injection-vulnerable code 36% of the time against 7% for the control. Pearce et al. ("Asleep at the Keyboard", IEEE S&P 2022) found about 40% of Copilot completions vulnerable across deliberately high-risk scenarios. Veracode's 2025 report ran the question across more than 100 models and found security flat across model generations.

The flatness is the tell. If the newest, largest model is not meaningfully safer than the older one, then "which model" is the wrong axis. The question worth asking is not whether the code is insecure but which parts fail and what closes the gap.

02

the experiment

Four models: Claude Opus 4.6, GPT-5.3-Codex, Gemini 3.1 Pro, and Kimi K2.5. Two tasks, both deliberately dense with security-critical surface: a Flask/SQLite login system with registration and authentication, and a REST API for user profiles with JWT auth. Two prompt conditions: a neutral prompt stating functional requirements only, and a security prompt that explicitly asked for SQL-injection protection, password hashing, input validation, secrets handling, and rate limiting. Four models times two tasks times two prompts is sixteen code samples, one run per cell.

Each sample was scored on six categories (SQL injection, password storage, input validation, XSS, secrets, auth/cookies), zero to two points each, twelve max. Zero meant the vulnerable pattern (string-concatenated queries, plaintext passwords, hardcoded secrets); two meant the secure one (parameterized queries, bcrypt or argon2, secrets in environment variables, httpOnly and Secure cookie flags). We then ran every sample through Semgrep on its default Python/Flask ruleset, and separately reviewed all sixteen by hand against the same checklist.

The honest framing comes before the results: n is sixteen, one generation per combination, and model output is non-deterministic. This is a within-study descriptive pattern, not a generalizable effect size. It is one of two strands. The other is a semester of shipping product at ToSee, which is qualitative, and which I keep separate from the experiment throughout, because conflating them is the easiest way to overclaim.

03

the inverted failure profile

Here is the per-category pass count, out of eight samples each:

Per-category pass count, out of eight samples each, under the neutral and security prompts
Category Neutral (/8) Security (/8)
sql injection protected8/88/8
password correctly hashed3/88/8
input validated2/88/8
xss protected1/88/8
secrets in env vars0/86/8
auth / cookies secured0/88/8

Read the neutral column top to bottom. SQL injection passes everywhere, then it falls off a cliff. Ranked as failure rates under the neutral prompt: hardcoded secrets 100%, insecure auth/cookies 100%, missing XSS protection 87.5%, missing input validation 75%, plaintext passwords 62.5%, SQL injection 0%. The one class everyone teaches passes in every sample. The classes nobody publishes, secrets and cookie flags, fail in all eight.

My interpretation of why: this tracks training-data representation, not model capability. Parameterized queries are over-represented in public code because SQL injection has been the canonical example for two decades, so the secure pattern is what the model has mostly seen. Secrets handling and httpOnly/Secure flags are configuration details that rarely show up in published snippets, so the model has mostly seen the insecure default. The per-category numbers are measured; the training-data explanation is interpretation, not something the experiment isolated. It is consistent with the data and the literature, and proven by neither.

The literature corroborates which classes recur. Fu et al. (ACM TOSEM, 2023/2024) found CWE-79 (XSS), CWE-330 (insufficient randomness), and CWE-94 (code injection) among the most common across 733 real Copilot snippets. Veracode (2025) reported XSS left undefended in 86% of relevant samples (it labels this CWE-80). Neither Fu et al. nor Veracode frames the result as an inversion driven by corpus representation. That framing is what our data adds.

04

security is a prompt-quality function

Add the security prompt and the picture changes almost completely. The average score across all samples rose from 5.875 to 11.625 out of twelve, close to a doubling. Two models, Claude Opus 4.6 and GPT-5.3-Codex, hit a clean 12/12 with the prompt. The thesis is explicit: that figure is "a descriptive measure of the difference between the two conditions in this experiment, and must not be interpreted as a statistically generalizable effect size." One run per cell. A pattern, not an effect size.

But look back at the table, secrets row, security column: 6/8. Even when explicitly asked to handle secrets correctly, two of eight samples still hardcoded them. The prompt is necessary and it is not sufficient. The reframing the thesis lands on is that the models can write secure code, they just do not unless asked, and even then not always.

One published result cuts against the easy reading. Shukla et al. (IEEE ISTAS 2025) found that iterative refinement, asking a model to improve its code across rounds, produced an overall 37.6% increase in critical vulnerabilities across the study's four prompting strategies (efficiency, feature, security, ambiguous) after five iterations. One well-formed security prompt helps in a single shot (our result); repeatedly leaning on the model to self-correct can degrade the code (theirs). The operational takeaway is the same either way: a human has to check the output between steps. The model does not verify itself.

05

the cheap model

We included Kimi K2.5 to test the assumption that a cheaper model would produce worse security. On the neutral prompt it scored highest of the four, 7/12, ahead of Claude Opus 4.6 at 6.5, with GPT-5.3-Codex and Gemini 3.1 Pro both at 5. The cheap model had the best unprompted defaults.

That is one sample per model, so the safe statement is narrow: the price-equals-security assumption did not hold here. What gives the cell weight beyond its own n is the convergence with the largest-scale evidence available. Veracode ran more than 100 models and found that newer and larger ones did not write more secure code; security stayed flat regardless of model size or training sophistication, with roughly 45% of samples introducing flaws. Recency and tier do not predict safety at their scale either. "Cheap models are as secure" is not a claim this experiment can make, and I am not making it. The assumption failed in the only place we tested it, and the direction agrees with the largest study that has asked.

06

what static analysis missed

This is the result that matters most for a small team. Across all sixteen samples:

Samples with vulnerabilities found, by detection method, across all sixteen samples
Method Samples with vulnerabilities found
semgrep (sast)2/16
manual review12/16

Semgrep, on its default ruleset, flagged two samples, and only for hardcoded secrets. It caught none of the plaintext passwords, missing validation, XSS, or insecure cookies. The manual checklist review found problems in twelve, including all eight neutral samples and the two security-prompt samples where secrets were still hardcoded.

The caveat is mandatory and the thesis states it: this is Semgrep with its default ruleset, not a verdict on static analysis in general. Schreiber and Tippe ran CodeQL across 7,703 AI-attributed files and surfaced 4,241 CWE instances across 77 types, though 87.9% of those files carried no CWE-mapped vulnerability at all, so detection rate is heavily a function of tool and ruleset. The 2/16 is not "SAST is useless." It is that the cheap, default SAST a small team actually reaches for, out of the box, will sail past most of what AI code gets wrong, and a five-minute human checklist will not. That is the reading a team without a security org needs to hear. Where tooling does excel is closing known classes fast once found: GitHub's Copilot Autofix data, vendor-reported from a public beta, claims SQL-injection remediation in 18 minutes against 3.7 hours manually. The gain is in verification tooling, not in trusting generation.

07

overtrust and the workflow that waves it through

The risk is not output quality in isolation. It is velocity times overtrust, inside a workflow that merges before anyone looks. I did not measure overtrust in our own team. We imposed the control, AI code was never treated as finished and was reviewed before merge, but we never measured overtrust or its absence in ourselves. The external evidence is strong on its own: Snyk's 2023 survey found only about 10% of professionals scan most of their AI output. Our contribution is not measuring that gap, it is the control built to sit inside it.

08

the framework gap

If the risk lives in the team's process rather than in the model, the obvious move is to reach for an existing secure-development framework. Every mature one presupposes an organization the small team does not have.

Secure-development frameworks and the organization each one assumes
Framework What it assumes
nist ssdf (sp 800-218)org-level roles and toolchain; a "prepare the organization" practice group
nist sp 800-218aaudience is producers and acquirers of AI models, not consumers of an assistant
owasp llm top 10secures the LLM in your product, not the developer-uses-an-LLM workflow
owasp samm v2a maturity ladder, not day-one controls
microsoft sdl (ai evolution, 2026)six standing organizational security functions

Two are worth spelling out. NIST SP 800-218A, by its own stated audience, is guidance for people building and acquiring AI models, not for the three-person shop consuming Claude through a CLI. The Cloud Security Alliance's Secure Vibe Coding Guide (Huang, April 2025) is the closest analog and worth citing as related work, but it is a generic individual checklist, not an agile-process-integrated, experiment-grounded framework.

The clearest datable sign the space is still open is OWASP's own. There is a proposal, filed as GitHub issue #826 in April 2026, for an "OWASP Top 10 for AI-Generated Code." It is unapproved and unmerged. A credible proposal for exactly this category, existing only as an open issue, dates how recently the gap was still vacant.

09

SAIF, briefly

So we built a framework for the team that has none of that substrate: two to eight people, agile, using AI tools daily, no dedicated security roles. SAIF (Secure AI-Integrated Framework) is five components, described as what a three-person team does on day one, not as a maturity model:

The five components of the SAIF framework and what each does
Component What it does
model routingroute tasks to models by type and security sensitivity, against a living model-profile catalogue
three-level securityeffort scales with risk: level 1 (auth, secrets, user data) makes the security prompt and full checklist mandatory; level 3 (styling, no user data) skips them
review checklistsix questions, one per experiment category, under five minutes per PR, blocks merge on any "no"
sprint workflowwires the above into planning, development, review, integration, and retrospective
adoption staircaselayer it in over weeks instead of all at once

The under-studied piece is the routing, and it needs a distinction the experiment did not. The four benchmarked models (Claude Opus 4.6, GPT-5.3-Codex, Gemini 3.1 Pro, Kimi K2.5) are not the same as the CLI tools we actually shipped with day to day (Claude Code, Codex CLI, Opencode). Most prior work tests one model in isolation, usually Copilot. The ToSee strand, qualitatively, is where deliberate per-task tool choice showed its value. One concrete decision: we routed debugging and reading an unfamiliar part of the codebase to Claude Code, because it was consistently strongest there, and the observed payoff was less time lost to wrong-headed edits on code the author had not written. The practical primary tool ended up being Opencode, because it runs multiple models in one terminal and the routing stops being a context-switch. That is informed practice from one deployment, not a proven result, and we found no external source that validates the routing practice. It is the contribution the controlled experiment could not produce, which is exactly why it is the least corroborated.

The necessary admission: SAIF is a proposed control, not a validated one. No other team has tested it. The adoption-staircase timeframes are recommendations from one case, not experience-confirmed standards. The whole thing is the prescription the literature points toward and does not build, which is what makes a concrete framework worth publishing even unvalidated. It does not make it proven.

10

limits

n=16, one run per cell, non-deterministic generation. A single team, a single company, React and Bun in the product and Python/Flask in the experiment. Static analysis and manual review only, no dynamic testing or pen-testing of running systems. The SAST result is specific to Semgrep's default ruleset. Productivity at ToSee was experienced qualitatively and never measured. Every productivity percentage in this genre belongs to someone else's study, and at least one cuts against the optimistic reading: METR's 2025 result found that allowing AI increased experienced open-source developers' task completion time by 19%, slowing them down. I will not imply we measured a speedup. This is analytic generalization, the kind you reason from to your own context, not statistical generalization. Take the numbers as indications of a pattern, then verify in your own stack.

11

where this goes

The through-line is short. The security failures in AI-generated code are not where the model is weak, they are where the training corpus is thin; and they are not closed by buying a better model, they are closed by a cheap process control: an explicit prompt for security-critical code, and a five-minute checklist a human runs before merge. The model-choice spread in our data, two points neutral and one with a prompt, was smaller than what one prompt bought, 5.75. Process is where the gain is.

Which lands on the thing I keep coming back to in security work. The hard part was never generating the code. It is verifying that the control worked, that the secret really is in an environment variable, that the cookie really has its flags, rather than trusting the thing that wrote it. You verify the control fired. You do not trust the generator.
